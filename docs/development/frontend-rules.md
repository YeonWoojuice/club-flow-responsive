# 프론트엔드 개발 규칙

> 이 문서는 `frontend/` 디렉터리의 구현과 앞으로 진행할 UI 개편에 적용합니다.
> 현재 API 계약과 인증 흐름을 유지하면서 화면 구조를 점진적으로 개선하는 것이 기본 원칙입니다.

## 0. 지시 우선순위와 기준 파일

작업 중 지시가 충돌하면 다음 순서를 따릅니다.

1. 현재 사용자가 명시한 요청과 제한 사항
2. 현재 백엔드 API 계약과 제품 요구사항
3. 이 문서의 프론트엔드 규칙
4. 프레임워크의 일반적인 관례

- 충돌 여부가 불분명하면 임의로 결정하지 않고 관련 파일을 확인한 뒤 사용자에게 알립니다.
- 문서 설명보다 현재 실행 코드와 API 계약이 달라졌다면 차이를 먼저 보고합니다.
- 현재 프론트엔드 작업의 기준 파일은 다음과 같습니다.

```text
frontend/src/router/AppRouter.tsx       현재 URL과 보호 경로
frontend/src/api/http.ts                HTTP, 인증 오류 이벤트, CSRF 처리
frontend/src/auth/AuthProvider.tsx      인증 상태 전이
frontend/src/types/                     서버 요청·응답 타입
docs/product/requirements.md            제품 기능과 상태 정책
```

## 1. 기술 기준

| 항목 | 기준 |
|---|---|
| 프레임워크 | React 19 함수형 컴포넌트 |
| 언어 | TypeScript 5.8 strict |
| 빌드 | Vite 7 |
| 라우팅 | React Router 7 |
| 스타일 | Tailwind CSS 4 + CSS Custom Properties |
| 테스트 | Vitest + Testing Library |
| 인증 | 서버 세션 + CSRF |
| 서버 상태 | 도메인 API 모듈 + 페이지 상태 |

- `any`와 `as any`를 사용하지 않습니다.
- 서버 응답 타입은 `src/types/`에 선언합니다.
- 새로운 전역 상태 도구는 실제로 여러 화면이 공유해야 하는 요구가 생긴 뒤 도입합니다.

## 2. 디렉터리 책임

```text
src/
├── api/         공통 HTTP 처리와 도메인 API
├── auth/        인증 상태, 보호 경로, 인증 오류 UI
├── components/  여러 화면에서 재사용하는 UI
├── pages/       URL 단위 화면과 화면 전용 상태
├── router/      라우트 선언
├── test/        테스트 공통 설정
├── types/       서버 응답과 요청 공유 타입
├── App.tsx
├── main.tsx
└── styles.css
```

- API 호출과 `fetch` 설정은 `src/api/`에 둡니다.
- 페이지에서만 사용하는 폼·필터·선택 상태는 해당 페이지에 둡니다.
- 두 화면 이상에서 같은 UI나 상태 표현을 사용하면 `components/` 또는 도메인 공통 파일로 분리합니다.
- 컴포넌트는 API를 직접 조합하지 않고 도메인 API 함수를 호출합니다.

## 3. 변경 원칙

- UI 개편은 기존 URL, API 요청 형식, 상태 전이 규칙을 임의로 바꾸지 않습니다.
- 기존 공통 컴포넌트와 `styles.css`의 CSS 변수를 먼저 재사용합니다.
- 한 번에 전체 화면을 재작성하지 않고 레이아웃, 공통 컴포넌트, 페이지 순으로 나눠 변경합니다.
- 현재 동작하는 비즈니스 흐름을 유지하면서 로딩·빈 화면·오류·처리 중 상태를 함께 개선합니다.
- 사용자가 커밋을 요청한 경우에만 커밋하며, 화면 구조 변경과 API·인증 로직 변경은 별도 커밋으로 분리합니다.

## 4. 인증 상태

인증 상태의 단일 기준은 `AuthProvider`입니다.

| 상태 | 의미 | 화면 처리 |
|---|---|---|
| `loading` | 최초 로그인 상태 확인 또는 오류 재시도 중 | 로딩 화면 |
| `authenticated` | 로그인 사용자 확인 완료 | 보호 화면 표시 |
| `anonymous` | 서버가 `401`을 반환한 미로그인 상태 | 로그인 화면 이동 |
| `error` | 최초 확인 중 500 또는 네트워크 오류 | 일시적 오류 화면과 재시도 |

### 인증 오류 처리

- `401`일 때만 사용자 정보를 비우고 `anonymous`로 변경합니다.
- `500` 또는 네트워크 오류를 로그아웃으로 간주하지 않습니다.
- 이미 `authenticated`인 상태에서 일시적 오류가 발생하면 기존 사용자와 인증 상태를 유지합니다.
- 기존 인증 상태를 유지한 오류는 `AuthErrorNotice`에서 안내하고 재시도할 수 있게 합니다.
- 최초 인증 확인이 실패하면 `AuthErrorScreen`을 표시하고 재시도할 수 있게 합니다.
- 인증 오류 분류는 `authFailure.ts`를 통해 처리하며 페이지마다 같은 판정을 반복하지 않습니다.

### 로그아웃

- 로그아웃 성공 후에만 `clear()`를 호출하고 로그인 화면으로 이동합니다.
- 로그아웃 실패 시 기존 인증 상태와 현재 화면을 유지합니다.
- 로그아웃 실패는 `ErrorToast`로 사용자에게 알립니다.
- 처리 중에는 로그아웃 버튼을 비활성화해 중복 요청을 막습니다.

## 5. API와 CSRF

- 모든 요청은 `apiRequest()`를 사용합니다.
- 서버 세션 쿠키를 위해 `credentials: "include"`를 유지합니다.
- JSON 본문이 있는 요청에는 `Content-Type: application/json`을 설정합니다.
- 서버 오류의 `message`가 있으면 사용자에게 표시하고, 없으면 공통 기본 문구를 사용합니다.
- `401`은 로그인 화면, 최종 `403`은 접근 가능한 동아리 목록으로 처리합니다.

### CSRF 토큰 규칙

- `POST`, `PUT`, `PATCH`, `DELETE` 등 쓰기 요청 전에 `/api/auth/csrf`에서 토큰을 가져옵니다.
- 발급받은 토큰은 공통 HTTP 모듈에서만 캐시합니다.
- 로그아웃이 끝나면 성공 여부와 관계없이 캐시한 CSRF 토큰을 초기화합니다.
- 쓰기 요청이 `403`이면 기존 토큰을 폐기하고 새 토큰으로 요청을 정확히 한 번만 재시도합니다.
- 재시도도 `403`이면 추가 재시도하지 않고 기존 전역 권한 오류 처리로 넘깁니다.
- `GET`, `HEAD`, `OPTIONS`는 CSRF 재발급·재시도 대상이 아닙니다.
- 페이지나 도메인 API에서 CSRF 헤더를 직접 만들지 않습니다.

## 6. 라우팅

```text
/login                                      Google 로그인
/auth/callback                              로그인 완료 후 동아리 분기
/clubs                                      내 동아리 목록
/clubs/new                                  동아리 생성
/clubs/:clubId/dashboard                    승인된 운영진 대시보드
/clubs/:clubId/generations                  학기 관리
/clubs/:clubId/applications                 지원자 목록
/clubs/:clubId/applications/new             수동 지원자 등록
/clubs/:clubId/applications/:applicationId  지원서 상세
/clubs/:clubId/members                      부원 목록
```

- 보호 페이지는 `RequireAuth` 아래에 둡니다.
- 접근 가능한 동아리가 없으면 `/clubs/new`로 이동합니다.
- 접근 가능한 동아리가 하나면 해당 대시보드, 여러 개면 `/clubs`로 이동합니다.
- 앱 내부 이동은 `Link`, `Navigate`, `useNavigate`를 사용합니다.
- `clubId`가 포함된 화면은 해당 동아리 접근 권한 실패를 전역 `403` 규칙에 따라 처리합니다.

## 7. UI 개편 규칙

> 아래 규칙은 다음 UI 개편부터 적용합니다. 현재 화면에 남아 있는 고정 사이드바와 고정 폭 목록은 개선 대상입니다.

### 레이아웃과 반응형

- 모바일 우선으로 작성하고 `sm`, `md`, `lg` 순서로 확장합니다.
- 고정 사이드바는 작은 화면에서 모바일 내비게이션으로 전환합니다.
- 작은 화면에서 본문보다 사이드바가 우선해서 폭을 차지하지 않게 합니다.
- 표와 다열 목록은 모바일 카드로 바꾸거나 `overflow-x-auto`와 최소 폭을 명시합니다.
- 상세 화면의 본문과 사이드 패널은 모바일에서 세로로 배치합니다.
- `h-screen`이 모바일 브라우저에서 문제가 되면 동적 뷰포트 단위(`dvh`)를 사용합니다.
- 페이지 기본 여백은 모바일과 데스크톱을 구분합니다. 예: `p-4 sm:p-6 lg:p-8`.

### 화면 상태

데이터를 조회하거나 변경하는 화면은 다음 상태를 명시적으로 처리합니다.

1. 로딩
2. 빈 결과
3. 사용자 입력 오류
4. API 오류와 재시도
5. 제출·변경 처리 중
6. 성공 후 목록 갱신 또는 화면 이동

- 처리 중인 버튼은 비활성화하고 문구를 변경해 중복 요청을 방지합니다.
- 학기 종료, 지원자 합격·불합격·취소, 데이터 삭제처럼 되돌릴 수 없거나 최종 상태를 만드는 동작에는 확인 절차를 둡니다.
- 오류를 단순히 콘솔에 남기지 않고 화면 안의 오류 영역 또는 토스트로 알립니다.
- 권한 오류, 인증 오류, 일반 API 오류를 같은 문구로 처리하지 않습니다.

### 공통 컴포넌트

- 앱 공통 레이아웃은 `AppLayout`을 기준으로 합니다.
- 전체 화면 로딩은 `LoadingScreen`을 사용합니다.
- 일시적 오류 알림은 `ErrorToast`를 사용합니다.
- 상태 배지·필터·버튼 패턴은 동작과 시각적 의미가 같고 향후 재사용 가능성이 있을 때 공통 컴포넌트로 분리합니다.
- 단순히 두 번 등장했다는 이유만으로 추상화하지 않습니다.
- 공통 컴포넌트는 특정 페이지 URL이나 도메인 요청에 과도하게 결합하지 않습니다.

### 스타일과 접근성

- 색상은 `styles.css`의 CSS Custom Properties를 우선 사용합니다.
- 상태는 색상만으로 표현하지 않고 텍스트를 함께 표시합니다.
- 아이콘만 있는 버튼에는 `aria-label`을 제공합니다.
- 현재 내비게이션에는 `aria-current="page"`를 사용합니다.
- 오류 메시지는 `role="alert"`, 비동기 알림은 적절한 `aria-live`를 사용합니다.
- 키보드 포커스가 보이도록 `focus-visible` 스타일을 유지합니다.
- 폼 입력은 시각적인 문구만 두지 않고 `label`과 연결합니다.

## 8. 테스트 규칙

다음 명령은 저장소 루트에서 실행합니다.

```bash
cd frontend
npm run lint
npm test
npm run build
```

- API 공통 처리, 인증 상태 전이, 권한 이동은 자동화 테스트를 작성합니다.
- CSRF 변경 시 다음 세 경로를 검증합니다.
  - 정상 토큰 첨부
  - 첫 `403` 후 재발급과 1회 재시도
  - 재시도 실패 시 추가 재시도 중단
- 인증 변경 시 다음 세 경로를 검증합니다.
  - `401`만 `anonymous` 처리
  - 초기 네트워크 오류 안내와 재시도
  - 로그인 후 500 오류에서도 기존 사용자 유지
- 로그아웃 변경 시 실패 토스트와 인증 상태 유지 여부를 검증합니다.
- UI 개편 후 주요 화면을 `360px`, `768px`, `1280px` 너비에서 확인합니다.
- 어떤 검증 너비에서도 페이지 전체에 가로 스크롤이 생기면 안 됩니다.
- 넓은 표는 페이지가 아니라 표 컨테이너 내부의 가로 스크롤만 허용합니다.
- 기능 변경 후 린트, 전체 테스트, 프로덕션 빌드를 모두 통과시킵니다.

## 10. 금지 사항

- `window.location`을 이용한 앱 내부 페이지 이동
- 페이지 컴포넌트에서 직접 `fetch` 옵션과 CSRF 헤더 작성
- OAuth access token 또는 ID token을 브라우저 저장소에 보관
- 500·네트워크 오류를 임의로 로그아웃 처리
- 쓰기 요청의 무제한 자동 재시도
- 처리 중 버튼을 활성화한 채로 두는 것
- `console.log` 커밋
- `any`, `as any`, 불필요한 타입 단언
- 인라인 스타일과 페이지별 색상 하드코딩
- 동작 검증 없이 공통 레이아웃을 한 번에 전면 교체
